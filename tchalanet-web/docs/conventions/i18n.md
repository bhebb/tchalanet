# i18n Convention

> Status: DRAFT v0.2  
> Scope: local fallback bundles, backend/bootstrap overrides, language switching, key taxonomy

## Rule

i18n is resolved by merging ordered local fallback bundles with runtime backend/bootstrap
overrides. Local bundles provide safe offline fallback. Backend/bootstrap messages win when
the same key exists.

## Placement

```text
apps/tch-portal/src/app/core/i18n/                     loader, merge, language state
apps/tch-portal/public/assets/i18n/{locale}/           local fallback bundles
libs/shared-config/src/lib/runtime/runtime-paths.ts    stable i18n configuration
```

Use `core/i18n` for language state, translate loading, merge rules, and language
switcher integration. Use public assets only for local fallback bundles.

`PORTAL_I18N_CONFIG` owns default/fallback languages, asset prefix/suffix, and the ordered
bundle list. `app.config.ts` wires this configuration into the loader; it must not duplicate
paths or bundle names.

This follows the ownership rules in `placement-guide.md`: i18n runtime is app `core`;
translation assets are runtime data, not feature code.

## Bundle Layout

Local fallback bundles are split by locale first, then by logical bundle:

```text
apps/tch-portal/public/assets/i18n/
  fr/
    common.json
    domain.json
    component.json
    surface-public.json
    surface-admin.json
    surface-platform.json
    surface-seller-terminal.json
    feature-auth.json
    feature-public.json
    feature-admin.json
    feature-platform.json
    feature-seller-terminal.json
  en/
  ht/
```

JSON is nested. Angular code does not know which physical file owns a key.

## Merge Order

Local bundles merge in configured order from generic to specific:

```text
common
domain
component
surface-public
surface-admin
surface-platform
surface-seller-terminal
feature-auth
feature-public
feature-admin
feature-platform
feature-seller-terminal
then backend/bootstrap overrides
```

Later bundles override earlier local bundles. Backend/bootstrap overrides win last.
Missing local files must not break rendering at runtime, but contract tests should fail when
a required locale bundle or key is missing.

Do not duplicate the same final key in multiple local bundles. A duplicate key is allowed
only for an intentional, documented override; otherwise move the key to its owning bundle.

## Taxonomy

Use these logical namespaces:

| Namespace | Purpose |
| --- | --- |
| `common.*` | Generic actions, states, validation, yes/no. No Tchalanet domain terms. |
| `domain.*` | Canonical Tchalanet vocabulary, entities, statuses, and shared business labels. |
| `component.*` | Reusable component labels and messages. |
| `surface.*` | Shell and navigation labels by surface. No page body copy. |
| `feature.*` | Page and workflow copy. |

Physical surface bundles own shell/navigation copy even when existing keys still use
legacy namespaces during migration:

| Bundle | Owns |
| --- | --- |
| `surface-public.json` | Public header, footer, skip links, public shell navigation. |
| `surface-admin.json` | Tenant/admin sidenav and private shell labels. |
| `surface-platform.json` | Platform sidenav and platform shell labels. |
| `surface-seller-terminal.json` | Seller-terminal sidenav and operational shell labels. |

Feature code still follows `feature-structure.md`: routes and components live under
`features/private/<surface>/<feature>` or the matching public/auth area. Translation files
remain centralized under `assets/i18n/{locale}` instead of being placed beside components.

`common.json` is reserved for the `common.*` namespace. Do not place `app.*`, `error.*`,
domain terms, component messages, shell copy, or feature labels in `common.json`.

Feature bundles must not own navigation labels. For example, public home section copy may
use `home.check_ticket.title`, but public navigation labels must use `surface-public`
keys such as `public.nav.check_ticket`; do not add `home.nav.check_ticket`.

Use domain keys for reusable business vocabulary even when the current screen is public.
For example, the reusable field label is `domain.ticket.field.publicCode`; the public
verification page owns its experience copy under `public.check.*`. Do not create a
feature-local pseudo-domain such as `public.ticket.*`.

Short reusable business statuses belong to `domain`, even when first introduced by a page.
For example, use `domain.result.status.CONFIRMED`; keep explanatory public-page messages
such as `public.results.detail_status.CONFIRMED.body` in the page namespace.
Reusable result field labels also belong to `domain`, for example
`domain.result.field.drawDateTime`, `domain.result.field.supportedSource`, and
`domain.result.field.publicReference`.

## Keys

Use stable dot keys. Prefer the new taxonomy for new keys:

```text
common.action.refresh
domain.draw.status.open
surface.admin.nav.draws
feature.adminDraws.title
```

Existing keys may remain temporarily during migration, especially when PageModel or backend
bootstrap payloads reference them. Record intentional aliases in the migration inventory and
remove them in a dedicated cleanup slice.

Naming follows `naming.md`: keys are searchable, predictable, and avoid creating a second
name for the same product concept.

Before adding a page-specific label, ask whether the text is generic or domain vocabulary.

Generic actions and states use shared `common.*` keys: `common.add`, `common.save`,
`common.edit`, `common.delete`, `common.cancel`, `common.view`, `common.details`,
`common.refresh`, `common.search`, `common.filter`, `common.resetFilters`,
`common.export`, `common.print`, `common.verify`, `common.confirm`, `common.archive`,
`common.activate`, `common.deactivate`, `common.block`, `common.unblock`.

Product nouns use `domain.entity.*`: `domain.entity.draw`, `domain.entity.seller`,
`domain.entity.sellerTerminal`, `domain.entity.odds`, `domain.entity.promotion`,
`domain.entity.ticket`, `domain.entity.game`, `domain.entity.result`, and their plural
forms when the visible label is only the noun.

Create a feature or surface key only when the visible wording is genuinely contextual
(`platform.tenants.action.create`, `admin.draws.empty.message`) or when an intentional
override is documented. Do not create two keys for the same displayed word.

## Runtime Paths

```text
local fallback: /assets/i18n/{lang}/{bundle}.json
backend/bootstrap: delivered by runtime bootstrap and merged into ngx-translate
```

Backend i18n catalog endpoints remain backend-owned. Browser calls stay relative so local
serve targets can proxy API requests.

## Product Language

- Platform/superadmin may use technical labels such as tenant, cache, audit, archive.
- Tenant admin should use business labels such as espace, vendeurs, tirages, limites,
  commissions.
- Seller terminal should use short operational labels such as vendre, tickets, verifier,
  reimprimer, mes stats, PIN.

Do not expose legacy internal terms such as cashier, odds, or seller-terminal in business
surfaces when a business label exists.

## Reference Migration

During migration, use an inventory script to compare:

- declared keys in merged local bundles;
- keys referenced in `.ts`, `.html`, routes, and PageModel fallback configs;
- declared but unused keys;
- referenced but missing keys.

Migrate references by surface, not by sweeping the entire app at once:

1. shared `common/domain/component`;
2. `surface-*` navigation and shell;
3. `feature-auth` and `feature-public`;
4. `feature-admin-*`;
5. `feature-platform-*`;
6. `feature-seller-terminal`.

## Anti-Patterns

Do not:

- fetch translation JSON directly from components;
- place feature translation files beside feature components;
- duplicate the same label under many namespaces without a migration reason;
- create navigation labels inside feature/page bundles (`home.nav.*`, page-local footer
  labels);
- create feature-local domain families such as `public.ticket.*` when the concept belongs
  to `domain.ticket.*` or the screen already has a page namespace such as `public.check.*`;
- create page-local copies of reusable business statuses such as `public.results.status.*`
  instead of `domain.result.status.*`;
- create page-local result field labels such as `public.results.table.date` or
  `public.results.draw_datetime` instead of `domain.result.field.*`;
- create page-specific copies of generic actions or shared domain nouns;
- put tenant-specific copy in local assets when it should be a backend override;
- let PageModel carry translation maps;
- block the app when override loading fails;
- concatenate translated fragments when one full sentence key is possible.
