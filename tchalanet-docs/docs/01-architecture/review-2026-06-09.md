# Revue d'architecture Tchalanet — 2026-06-09

Audit transversal des cinq slices (`tchalanet-server`, `tchalanet-web`, `tchalanet-mobile`,
`tchalanet-edge-service`, `tchalanet-infra`) + le slice docs. Lecture statique : routeurs
`AGENTS.md`, manifestes, structure des sources, compose/env, OpenSpec en cours.
Aucun code modifié. Chaque section se termine par une liste de suivi actionnable.

---

## 1. Vue d'ensemble

| Slice | Stack | État architectural |
| --- | --- | --- |
| server | Java 25, Spring Boot 4.0.6, Maven multi-modules | Couches strictes (common→catalog→platform→core→features) + ArchUnit. Solide. |
| web | Angular 21.2, Nx 22.7, pnpm 10 | App `tch-portal` + libs (api, page-model, ui, widgets, web, shared-config). Boundaries ESLint actives. |
| mobile | Flutter (Dart sdk ^3.11.5), Riverpod, go_router | Feature-first (`app/core/design_system/features`). Couverture de tests quasi nulle. |
| edge-service | Node 24, Fastify 5, TypeScript | Modules delivery/health/messages/ping. Périmètre clair (notifications sortantes Brevo/Twilio/Slack). |
| infra | Docker Compose, Traefik v3.7, PG 18.4, Keycloak 26.6.2 | Compose modulaire + envs (common/dev/staging/prod). Collision de nommage « edge ». |

Le modèle slice-first (routeurs AGENTS.md, OpenSpec par projet, skills) est cohérent et bien
appliqué. Les écarts relevés sont surtout du **drift documentaire**, de l'**hygiène de dépôt**
et quelques **pièges opérationnels** côté infra.

## 2. Constats transverses

### 2.1 Drift `VERSIONS.md` (source de vérité non synchronisée)

`VERSIONS.md` se déclare source de vérité mais diverge des manifestes réels :

- Spring Boot : **4.0.3 annoncé vs 4.0.6 dans `tchalanet-server/pom.xml`**.
- Mobile : `go_router` ^17.3.0 annoncé vs ^17.2.3 réel ; `flutter_secure_storage` ^10.3.1 vs
  ^10.2.0 ; Dart 3.12.0 annoncé vs contrainte sdk ^3.11.5 dans `pubspec.yaml`.
- **Section « 4) Edge service » absente** (la numérotation saute de 3 à 5) alors que
  `tchalanet-edge-service/package.json` est listé comme source de vérité.
- Images présentes dans `envs/common/compose.env` mais absentes du tableau infra :
  **Unleash 7.4.0, Meilisearch v1.11, Doppler CLI 3.75.1**.

Cause d'erreur probable : un agent ou un humain qui se fie à VERSIONS.md prendra des décisions
(compatibilité, upgrade, debug) sur des versions fausses — exactement ce que la règle du fichier
veut empêcher.

### 2.2 Double lockfiles npm + pnpm

`tchalanet-web` et `tchalanet-edge-service` contiennent **à la fois `package-lock.json` et
`pnpm-lock.yaml`** (les deux trackés). Le `packageManager` déclaré est pnpm. Cause d'erreur
probable : installs divergentes selon l'outil utilisé (CI vs local vs agent), résolutions de
dépendances incohérentes, et confusion des outils d'audit.

### 2.3 Hygiène de dépôt (fichiers trackés qui ne devraient pas l'être)

- `tchalanet-web/web-backup/` : **l'ancienne app complète (`tchalanet-portal`) est trackée**,
  y compris un `.env` (contenu bénin vérifié, mais le pattern est dangereux) et tous ses assets.
- ~12 captures d'écran PNG à la racine de `tchalanet-web/`.
- `tchalanet-server/` : `build_output.txt`, `test_curl.sh`, `dev-local.crt`,
  `local-truststore.jks`, `tchalanet-local-truststore.jks` trackés (la clé `dev-local.key`
  n'est pas trackée — bien).
- `tchalanet-infra/Makefile.bckup` tracké.

### 2.4 Java 25 vs Java 21

`tchalanet-server` exige Java ≥ 25 (enforcer), mais
`tchalanet-infra/keycloak/tchalanet-keycloak-provider/pom.xml` reste en **Java 21**.
Déjà noté dans VERSIONS.md, toujours non résolu. Acceptable (le provider tourne dans la JVM
de Keycloak), mais à homogénéiser ou à documenter comme choix délibéré (ADR).

### 2.5 Backlog OpenSpec non archivé

Beaucoup de changes ouverts : ~15 côté server, ~11 côté web, 9 à la racine, 5 mobile, 2 edge.
Plusieurs semblent livrés (ex. `public-shell-v1-header-footer`, `md3-theme-foundation`).
Cause d'erreur probable : un agent chargeant un change « actif » mais déjà livré travaille sur
une spec périmée.

### Suivi transverse

- [ ] Resynchroniser `VERSIONS.md` : Spring Boot 4.0.6, versions mobile réelles, ajouter §4 Edge service, ajouter Unleash/Meili/Doppler au tableau infra.
- [ ] Supprimer `package-lock.json` de `tchalanet-web` et `tchalanet-edge-service` ; ajouter un garde-fou (`.npmrc` ou CI) interdisant npm.
- [ ] Supprimer `web-backup/` du dépôt (garder un tag/branche d'archive si besoin).
- [ ] Purger les fichiers parasites : PNGs racine web, `build_output.txt`, `test_curl.sh`, `Makefile.bckup` ; compléter `.gitignore`.
- [ ] Statuer sur les truststores/certs dev trackés (les régénérer via script plutôt que les versionner).
- [ ] Passe d'archivage OpenSpec sur les 5 slices (déplacer les changes livrés vers `archive/`).
- [ ] Décider Java 21→25 pour le keycloak-provider (PR dédiée + build complet) ou ADR « reste en 21 ».

## 3. tchalanet-server

### Remarques

- Les couches `common/catalog/platform/core/features` (non-negotiables) correspondent bien aux
  modules Maven (`tchalanet-common`, `-catalog`, `-platform`, `-core`, `-features`, `-app`) :
  la structure de modules rend les violations de dépendances impossibles à la compilation. Bon point.
- Tests ArchUnit présents dans `tchalanet-app` (`PageModelArchTest`, `SecurityArchTest`,
  `TimezoneEnforcementArchTest`, `FlywayAuditAlignmentArchTest`) — mais **pas de test ArchUnit
  explicite des règles intra-module** (ex. `platform.<a>.internal` ↛ `platform.<b>.internal`,
  `core` ↛ `platform.internal`). Ces règles-là ne sont pas garanties par le découpage Maven.
- Migrations Flyway : 34 fichiers dans `tchalanet-app/.../db/migration` mais numérotés jusqu'à
  `V222` — la numérotation est éparse (migrations supprimées/squashées ?). Si un environnement a
  déjà appliqué des versions disparues, `flyway validate` cassera ou masquera un drift de schéma.
- Package racine `com.tchalanet.server` (le nom historique `net.tchala` n'existe plus — OK).

### Causes probables d'erreurs

- Numérotation Flyway éparse + règle pré-go-live « demander avant tout nouveau `V*.sql` » :
  risque de collision de numéros entre branches.
- Provisioning tenant incomplet : la création de tenant ne seed ni plan ni catalogue
  (cf. gaps connus) — toute démo/onboarding sans seed manuel échoue de façon non évidente.
- Re-import du realm Keycloak sans restart du serveur ⇒ cache JWKS/issuer périmé côté API
  (symptôme : 401 inexpliqués après régénération du realm).

### Suivi server

- [ ] Ajouter un `LayerDependencyArchTest` couvrant les règles `10-non-negotiables.md` non garanties par Maven (platform.internal croisés, core→platform.api uniquement).
- [ ] Documenter (ou vérifier) l'historique Flyway : pourquoi 34 fichiers pour V1→V222 ; confirmer que tous les environnements valident (`flyway validate`).
- [ ] Compléter le provisioning tenant (seed plan/catalogue ou parcours REST documenté de bout en bout).
- [ ] Retirer `build_output.txt`, `test_curl.sh` et les truststores du versioning (cf. transverse).

## 4. tchalanet-web

### Remarques

- **Drift du routeur** : `tchalanet-web/AGENTS.md` référence `apps/tchalanet-portal/` alors que
  l'app réelle est `apps/tch-portal/` (l'ancienne vit dans `web-backup/`). Un agent suivant le
  routeur explore le mauvais arbre.
- Découpage libs sain (`api`, `page-model`, `widgets`, `ui/{components,styles,theme}`,
  `shared-config`) avec `@nx/enforce-module-boundaries` actif. La lib **`libs/web` a un nom trop
  vague** — son rôle n'est pas devinable depuis le nom (contrairement aux autres).
- L'app distingue `features/{public,admin,platform,dashboard,dev}` — aligné avec le travail
  public/admin en cours (PageModel renderer). `nx-welcome.ts` traîne encore dans `src/app`.
- Pipeline thème M3 outillé (`theme:generate`, `tokens:generate`) — bon point.

### Causes probables d'erreurs

- Double lockfile (cf. 2.2) : `npm install` accidentel ⇒ `node_modules` incohérent avec pnpm.
- Doc routeur pointant vers une app inexistante ⇒ patchs appliqués dans `web-backup/` par erreur
  (il est tracké, donc modifiable sans alerte).

### Suivi web

- [ ] Corriger `AGENTS.md` : `apps/tchalanet-portal/` → `apps/tch-portal/`.
- [ ] Supprimer `web-backup/` et les PNGs racine (cf. transverse).
- [ ] Renommer ou documenter `libs/web` (README + tags Nx explicites).
- [ ] Supprimer `nx-welcome.ts`.
- [ ] Vérifier que les tags `enforce-module-boundaries` couvrent les nouvelles libs (`page-model`, `widgets`) et pas seulement `scope:shared`.

## 5. tchalanet-mobile

### Remarques

- Structure conforme au routeur : `lib/app`, `lib/core` (config/i18n/network/settings/storage/theme),
  `lib/design_system`, `lib/features/{auth,cashier,draw,pos}`. Feature-first propre.
- **Couverture de tests quasi nulle : 1 seul fichier dans `test/`** pour 4 features dont POS
  (parcours de vente, donc critique métier).
- Drift pubspec vs VERSIONS.md (cf. 2.1).
- Deux fichiers IDE (`tchalanet.iml`, `tchalanet_mobile.iml`) à la racine.

### Causes probables d'erreurs

- Règle « offline submissions = pending, jamais confirmed » : sans tests, aucune garantie qu'une
  régression ne fasse pas passer une vente offline pour confirmée — c'est l'invariant mobile le
  plus risqué du produit.
- Binding terminal / preuve device (`terminal-device-proof-client` en cours) : dépend du
  `terminal_binding` côté server (bug de hash seed déjà corrigé en V217) — tout changement de
  format de preuve doit être synchronisé via OpenSpec racine, sinon ventes REJECTED.

### Suivi mobile

- [ ] Établir un socle de tests : au minimum les états pending/confirmed du flux POS offline et le binding terminal.
- [ ] Aligner `pubspec.yaml` ↔ VERSIONS.md (une seule vérité).
- [ ] Retirer les `.iml` du versioning si trackés.
- [ ] Documenter le contrat de preuve device dans un change OpenSpec racine (mobile + server).

## 6. tchalanet-edge-service

### Remarques

- Périmètre net : service Fastify de remise de messages (Brevo email, Twilio SMS, Slack webhook),
  modules `delivery`, `messages`, `health`, `ping`. Pas de logique métier — conforme.
- **Renommage `notification` → `messages` en cours** (`p1-rename-notification-route-to-messages`
  + change racine `openspec-server-edge-communication-rename`) : tant que server et edge ne sont
  pas livrés ensemble, les routes peuvent diverger.
- 3 fichiers de tests seulement ; pas de section VERSIONS.md (cf. 2.1) ; double lockfile (cf. 2.2).
- `@types/node` ^22 alors que le runtime cible est Node 24 (mineur, mais typings désalignés).

### Causes probables d'erreurs

- Rename de routes partiellement déployé ⇒ le server poste sur `/notification*` pendant que
  l'edge n'expose plus que `/messages*` (ou l'inverse) : notifications silencieusement perdues
  si le retry/idempotence côté server ne couvre pas les 404.
- Healthcheck compose sur `/ready` : vérifier que ce endpoint existe bien dans `modules/health`
  (un mismatch laisse le conteneur unhealthy et Traefik le sort du routage).

### Suivi edge-service

- [ ] Terminer et déployer le rename notification→messages de façon atomique (server + edge dans la même fenêtre), avec compat ascendante temporaire si besoin.
- [ ] Ajouter la section « 4) Edge service » dans VERSIONS.md (Node 24, Fastify 5, Brevo/Twilio/Slack).
- [ ] Supprimer `package-lock.json` ; passer `@types/node` en ^24.
- [ ] Étoffer les tests : un test par canal de delivery (mock Brevo/Twilio/Slack) + contrat des routes `messages`.
- [ ] Vérifier la cohérence healthcheck (`/ready`) entre Dockerfile/compose et le module health.

## 7. tchalanet-infra

### Remarques

- Compose bien modulaire (un fichier par service, index, `envs/{common,dev,staging,prod}`),
  images épinglées (pas de `:latest`), logging bornée, healthchecks systématiques. Bon socle.
- **Collision de nommage « edge » — l'écart le plus dangereux du slice** :
  - `docker-compose-edge.yml` = **Unleash Edge** (proxy feature flags), service key `unleash`,
    variables `EDGE_IMAGE_BASE`/`EDGE_PORT`/`EDGE_CONTAINER`, image `ghcr.io/tchalanet/edge` ;
  - `docker-compose-edge-service.yml` = **le Fastify edge-service** (notifications), variables
    `TCH_EDGE_IMAGE`/`TCH_EDGE_TAG` ;
  - et `DOCKER_NETWORK_EDGE` désigne un **réseau** Traefik.
  Trois sens différents pour « edge » dans le même dossier.
- Défauts piégeux dans `envs/common/compose.env` : `DOCKER_NETWORK_EDGE=edge-prod` /
  `back-prod` alors que `ENV=dev` dans le même fichier ; `IMAGE_TAG=stg-20251116-2` (tag de
  staging de novembre 2025) comme défaut global.
- `EDGE_IMAGE_BASE` n'est pas défini dans `compose.env` (seulement en défaut inline
  `ghcr.io/tchalanet/edge`) alors que les autres images y sont centralisées.

### Causes probables d'erreurs

- Un opérateur qui `up` le mauvais fichier « edge » (flags vs notifications) ou qui définit
  `EDGE_*` en croyant configurer le Fastify — debugging long garanti.
- Démarrage avec les défauts de `common/compose.env` sans override d'env ⇒ conteneurs dev
  attachés à des réseaux nommés `*-prod` et image API au tag staging périmé.
- Connus (mémoire de session, à traiter) : `KC_HOSTNAME` doit inclure le port externe —
  staging/prod doivent aligner `keycloak.env` + `TCH_KC_ISSUER` + `JWT_ISSUER_URI` au prochain
  déploiement ; re-import realm KC nécessite un restart serveur (cache) ; `make rebuild-api`
  n'est pas fiable (le rebuild+recreate manuel l'est).

### Suivi infra

- [ ] Renommer pour lever l'ambiguïté : `docker-compose-edge.yml` → `docker-compose-unleash-edge.yml` (+ variables `UNLEASH_EDGE_*`), réserver « edge-service » au Fastify ; mettre à jour `docker-compose.index.md`.
- [ ] Nettoyer `envs/common/compose.env` : défauts réseau neutres (`edge-${ENV}`), retirer le défaut `IMAGE_TAG` périmé (le rendre obligatoire par env), centraliser `EDGE_IMAGE_BASE`.
- [ ] Appliquer la migration KC_HOSTNAME+port sur staging/prod (keycloak.env + TCH_KC_ISSUER + JWT_ISSUER_URI) au prochain déploiement.
- [ ] Fiabiliser `make rebuild-api` (reprendre la séquence build+recreate documentée) et documenter le restart obligatoire après re-import de realm.
- [ ] Supprimer `Makefile.bckup`.
