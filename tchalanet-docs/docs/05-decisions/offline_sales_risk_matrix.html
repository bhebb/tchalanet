Tchalanet — Matrice des risques Offline Sales

Schéma de conception pour permettre la vente hors ligne en Haïti tout en limitant les fraudes, les dépassements de limites, les ventes après cutoff, les faux tickets, les problèmes de login offline et les conflits de synchronisation.

Offline normal, online optimal Serveur = source de vérité Payout offline interdit Snapshot signé obligatoire Sync idempotente par batch

Légende de criticité

Critique
Perte financière, fraude directe, ticket payable invalide.
Élevé
Risque opérationnel majeur, litige client, abus vendeur.
Moyen
Incohérence, besoin de review, impact contrôlable.
Faible
Risque surtout UX/support ou facilement récupérable.

Flow sécurisé cible

1. Online bootstrapLogin serveur, terminal connu, session ouverte, outlet autorisé.
2. Snapshot signéDraws, cutoff, pricing, limites, receipt config, validUntil.
3. Vente offline bornéeTickets locaux immutables, séquence monotone, reçu OFFLINE_PENDING.
4. Sync batchBatch idempotent, validation serveur ticket par ticket, détection gaps/replay.
5. Accept / reject / reviewCash session mis à jour uniquement pour tickets acceptés.

Schéma coloré des risques principaux

Critique

Fraude de transaction

Vendeur crée, modifie ou rejoue des tickets hors contrôle serveur.

  • Ticket payload signé
  • QR + hash court imprimé
  • Idempotency key unique
  • Revalidation serveur obligatoire
Critique

Vente après cutoff

Modification de l’heure device ou vente tardive sans connexion.

  • cutoffAt serveur dans snapshot
  • validUntil court
  • clock drift detection
  • sync refuse ou met en review
Critique

Payout d’un ticket non validé

Client réclame un gain avant que le ticket offline soit accepté.

  • Payout offline interdit
  • syncStatus doit être ACCEPTED
  • Reçu indique non payable avant validation
  • Public verify affiche PENDING/UNKNOWN
Élevé

Dépassement des limites

Vente massive offline au-delà du plafond par ticket/session/outlet.

  • maxOfflineTicketAmount
  • maxOfflineTotalAmount
  • maxOfflineTicketCount
  • Agrégation serveur par terminal/outlet/seller
Élevé

Login offline abusif

Compte révoqué, PIN partagé, terminal volé ou session expirée.

  • Offline token signé et court
  • PIN/biométrie locale
  • Device binding
  • Revocation check à la sync
Élevé

Sync sélective

Le vendeur synchronise seulement certains tickets.

  • localSequence monotone
  • Gaps détectés côté serveur
  • Sync append-only par terminal
  • Batch complet requis selon policy
Moyen

Session fermée pendant offline

Admin ferme la session/journée pendant que le vendeur continue.

  • salesSessionId dans snapshot
  • validUntil court
  • createdAtDevice comparé à closedAt
  • Review si zone grise
Moyen

Ancien pricing / odds

Vente avec une version de prix obsolète.

  • pricingSnapshotId obligatoire
  • oddsSnapshot par ligne
  • version signée
  • Expiration courte
Faible

Litige UX client

Le client croit que le ticket offline est définitif.

  • Reçu très explicite
  • Statut OFFLINE_PENDING
  • Dernière sync connue
  • Vérification publique claire

Actions offline permises / interdites

Permis offline

  • Login offline avec token/snapshot valide
  • Consulter le snapshot signé
  • Vendre dans l’enveloppe autorisée
  • Imprimer / réimprimer un ticket local
  • Annuler localement un ticket non synchronisé selon policy
  • Lister les tickets locaux
  • Préparer et envoyer une sync batch

Interdit offline

  • Payout / paiement de gain
  • Vente hors snapshot ou après validUntil
  • Vente sans session ouverte dans le snapshot
  • Vente sur draw/game/pricing absent du snapshot
  • Override admin, correction résultat, changement limites
  • Fermeture journée outlet ou session finale
  • Validation définitive ticket

Architecture cible

Mobile / Terminal

Stockage local chiffré, séquence monotone, signature ticket, impression OFFLINE_PENDING.

core.terminal

État terminal, clé terminal, syncState, lastSeen, verrouillage, autorisation offline.

core.sales

OfflineSalesCapability, SyncOfflineSalesCommand, TicketSyncStatus, SaleOrigin, risk flags.

core.session / outlet

Session obligatoire, cash summary, salesBlocked, fermeture journée, réconciliation.

Le domaine sales décide si une vente offline est acceptée. Le terminal fournit l’état opérationnel, mais ne devient jamais la source de vérité métier.
