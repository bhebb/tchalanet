# Skill — Testing Strategy

Use this skill for every Tchalanet testing task before selecting a slice-specific skill.

## Purpose

Choose the smallest test level that proves the risk, without turning all tests into slow e2e suites.

## Test levels

### Unit tests

Use for local rules and deterministic logic:

- domain policies and calculators;
- application handlers with fake ports;
- mappers and assemblers when mistakes are costly;
- frontend pure functions, stores/selectors, guards, interceptors;
- mobile providers/state notifiers, DTO parsing, validators.

Rules:

- fast;
- no network;
- no real database;
- no full Spring/Angular/Flutter runtime unless needed;
- each test name states the rule being proven.

### Integration tests

Use for wiring and framework boundaries:

- Spring MVC/security/controller validation;
- CommandBus/QueryBus registration;
- persistence adapters, Flyway, Testcontainers, RLS;
- idempotency aspects and event consumers;
- Angular runtime wiring, interceptors, PageModel bootstrap consumption;
- Flutter secure storage abstraction, Dio interceptors, terminal-binding client.

Rules:

- test one integration seam at a time;
- avoid duplicating business unit tests;
- prefer targeted slice tests over full app boot.

### E2E tests

Use for user-visible critical journeys:

- tenant onboarding;
- cashier onboarding and terminal binding;
- session open/close;
- sell ticket;
- public ticket verification;
- payout execution;
- permission denied after capability removal;
- multi-tenant isolation.

Rules:

- run through public API/UI entrypoints;
- use realistic fixtures;
- never replace unit tests;
- do not mix performance/load tests with e2e.

### Contract tests

Use when frontend/mobile depends on backend shapes:

- ApiResponse wrapper;
- ProblemDetail errors;
- PageModel payloads;
- ActionItem and NavigationDestination;
- receipt/print payloads.

## Always answer before writing tests

1. What risk is being tested?
2. Which level is the smallest sufficient level?
3. What fixtures are needed?
4. Which dependencies are fake, mocked, or real?
5. Which command proves the tests pass?

## Do NOT test

- Framework code (Spring Boot auto-config, Angular DI, Flutter widgets).
- Trivial getters/setters with no logic.
- Private methods directly (test through public API).
- Implementation details (internal state, method call order).
- Code generated by tools (OpenAPI clients, Lombok builders, Prisma models).
- The same business rule at multiple levels (test once at the right level).

## Anti-patterns

- Adding e2e for logic that should be a unit test.
- Testing private methods directly.
- Mocking the class under test.
- Asserting implementation details instead of observable behavior.
- Using random data that makes failures non-reproducible.
- Ignoring security/RLS/idempotency for critical endpoints.
