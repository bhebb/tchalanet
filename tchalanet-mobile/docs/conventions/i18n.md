# Convention — i18n mobile

**Scope** : tous les textes visibles utilisateur
**Status** : normative

## Locales

| Locale | Rôle |
| --- | --- |
| `ht` | défaut et fallback |
| `fr` | supportée |
| `en` | supportée |

Résolution au démarrage : locale sauvegardée, puis locale device supportée, sinon
`ht`.

La préférence de locale n'est pas sensible. Elle est persistée par le
`LocaleRepository` via `shared_preferences`, initialisé avant `runApp`, afin d'éviter
un flash dans une langue différente au démarrage.

## Règles

- Tout titre, label, hint, tooltip, message, erreur et description d'accessibilité
  passe par une clé i18n.
- Chaque clé locale existe dans `ht`, `fr` et `en`.
- Les fichiers locaux suivent la convention commune web/mobile :
  `assets/i18n/{locale}/{bundle}.json`.
- Les bundles physiques sont du JSON imbriqué. Le runtime mobile fusionne les bundles
  puis les aplatit en `Map<String, String>`.
- Les nouvelles clés utilisent la taxonomie commune : `common.*`, `domain.*`,
  `component.*`, `surface.*`, `feature.*`.
- Les widgets consomment le bundle i18n global; aucun provider i18n par feature.
- Routes, headers HTTP, clés storage et codes techniques ne sont pas traduits.
- Les messages backend utilisent `messageKey`, paramètres et fallback.
- Ticket/PDF/print restent traduits côté backend.
- Le changement de locale met à jour toute l'application sans redémarrage.
- Flutter 3.44 ne fournit pas de bundle Material/Cupertino `ht`. Le delegate mobile
  explicite garde les contrôles framework fonctionnels avec un fallback français
  temporaire; les textes Tchalanet restent obligatoirement en créole.

## Bundles locaux

Ordre de fusion local :

```text
common
domain
component
surface-admin
surface-platform
surface-seller-terminal
feature-auth
feature-public
feature-admin
feature-platform
feature-seller-terminal
```

Les bundles les plus spécifiques gagnent sur les bundles génériques. Les overrides runtime
issus du bootstrap gagnent toujours après les bundles locaux.

Les fichiers peuvent rester vides (`{}`) quand une surface n'a pas encore de textes mobile,
mais ils doivent exister dans `ht`, `fr` et `en` pour garder le contrat stable.

## Interdit

- texte utilisateur hardcodé dans Views, ViewModels, Repositories ou Services ;
- français utilisé implicitement comme fallback ;
- affichage direct de `exception.toString()` à l'utilisateur.
- fichiers i18n placés à côté des features au lieu du bundle global.
