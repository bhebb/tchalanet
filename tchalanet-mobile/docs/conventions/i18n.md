# Convention — i18n mobile

**Scope** : `tchalanet-mobile/`
**Status** : normative

---

## 1. Principe : local d'abord, backend enrichit

```
assets/i18n/<locale>.json   ← toujours disponible (offline, cold start)
        +
GET /i18n/overrides?locale= ← override backend (après auth, fail-safe)
        ↓
I18nBundle.mergeWith(overrides)
        ↓
i18nBundleProvider           ← consommé par les widgets
```

**Règle fondamentale** : en cas de clé présente des deux côtés, **le backend gagne**.

---

## 2. Structure des clés

Format : `<feature>.<écran>.<élément>`

```
app.name
auth.login.title
auth.login.subtitle
auth.login.button
auth.login.loading
auth.logout
auth.forbidden.title
auth.forbidden.message
auth.forbidden.back
pos.dashboard.title
pos.session.card_title
pos.session.user
pos.session.name
pos.session.tenant
pos.session.roles
pos.session.no_roles
pos.session.token_expiry
common.loading
common.error.network
common.error.timeout
common.error.unknown
common.retry
```

**Règles :**
- Tout caractère alphanumérique + underscore + point.
- Pas de majuscules dans les clés.
- Les nouvelles clés sont ajoutées **dans tous les fichiers locaux** en même commit.
- Le fallback d'une clé manquante est la clé elle-même (comportement de `I18nBundle.translate`).

---

## 3. Locales supportées

| Locale | Fichier | Statut |
|---|---|---|
| `fr` | `assets/i18n/fr.json` | Défaut |
| `en` | `assets/i18n/en.json` | Secondaire |

- La locale active est détectée depuis `PlatformDispatcher.instance.locale`.
- Si la locale du device n'est pas supportée, `fr` est utilisé.
- Ajouter une nouvelle locale = ajouter un fichier JSON + déclarer dans `_localBundleProvider`.

---

## 4. Providers Riverpod

```dart
// locale active
ref.read(localeProvider.notifier).setLocale('en');
ref.watch(localeProvider); // 'fr' | 'en'

// bundle mergé (sync, safe — empty si pas encore chargé)
final bundle = ref.watch(i18nBundleProvider);
bundle.translate('auth.login.title'); // → "Se connecter"
bundle.translate('clé.inconnue');     // → "clé.inconnue" (fallback = clé)

// bundle async (si vous avez besoin de l'état loading/error)
ref.watch(i18nProvider); // AsyncValue<I18nBundle>
```

**Règle** : les widgets utilisent `i18nBundleProvider` (synchrone). Le provider async `i18nProvider` est réservé au bootstrap ou aux cas où l'état loading doit être affiché.

---

## 5. Utilisation dans les widgets

```dart
class LoginPage extends ConsumerWidget {
  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final t = ref.watch(i18nBundleProvider);

    return Text(t.translate('auth.login.title'));
  }
}
```

**Interdit :**
- Chaînes UI hardcodées pour du texte utilisateur (messages, labels, titres).
- Appel direct à `rootBundle` depuis un widget.
- Créer un provider `i18n` par feature — un seul `i18nBundleProvider` global.

**Autorisé :**
- Hardcoder des valeurs purement techniques (noms de routes, clés de storage, codes d'erreur).
- Chaînes dans les tests unitaires (les tests ne chargent pas les assets).

---

## 6. Backend overrides

Endpoint (à confirmer avec le backend) :

```
GET /api/v1/i18n/overrides?locale=fr
Authorization: Bearer <access_token>

Response:
{
  "locale": "fr",
  "translations": {
    "pos.dashboard.title": "Mon POS Tenant"
  }
}
```

- L'appel est fait par `I18nService.fetchOverrides(locale)`.
- En cas d'erreur réseau ou de 4xx/5xx, `I18nOverrides.empty` est retourné silencieusement.
- Les overrides sont rechargés à chaque changement de locale (`FutureProvider.family` avec `ref.watch(localeProvider)`).

---

## 7. Ajout d'une clé

1. Ajouter la clé dans `assets/i18n/fr.json` **et** `assets/i18n/en.json`.
2. Utiliser la clé dans le widget via `t.translate('ma.nouvelle.cle')`.
3. Ne pas créer de constantes Dart pour les clés — la clé string est la source de vérité.

---

## 8. Changement de locale à chaud

```dart
// Dans un widget (ex: sélecteur de langue)
ref.read(localeProvider.notifier).setLocale('en');
// → déclenche le rechargement automatique de i18nProvider
// → i18nBundleProvider met à jour tous les widgets qui l'écoutent
```
