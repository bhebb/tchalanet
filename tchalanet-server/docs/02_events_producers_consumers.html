Tchalanet Offline/Sales Events Map

Carte des événements levés, producteurs probables, consommateurs probables et règles de listeners pour éviter les effets de bord cross-domain.

1. Principe événementiel

Les events métier sont publiés après commit. Les soumissions techniques offline ont leurs propres events et ne doivent pas alimenter les stats Sales.

Events Sales

Seulement pour tickets officiels acceptés ou transitions métier du ticket.

Events Offlinesync

Pour audit, sync, risque, review. Ils ne sont pas des ventes officielles.

2. Producteurs et consommateurs probables

Matrice cible pour vérifier qu’un nouveau flow n’introduit pas d’écriture cross-domain directe.

3. Règles d’écouteurs

À appliquer à chaque listener.

1

Idempotence

Le listener doit pouvoir recevoir le même event deux fois sans double impact.

2

Transaction séparée

Le domaine consommateur écrit dans sa propre transaction.

3

Tenant context

L’event transporte tenantId et IDs nécessaires. Les thread hops doivent rebinder le contexte.

4

Pas d’infra cross-domain

Un listener exécute une command locale via CommandBus, il n’appelle pas les repositories d’un autre domaine.
