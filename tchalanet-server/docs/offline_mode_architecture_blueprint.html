Tchalanet Offline Sales Blueprint

Page de conception fonctionnelle et technique pour le mode offline : composants, classes, séquences, responsabilités par feature, impacts inter-domaines et ordre de développement.

Décision structurante

✓

Une vente offline n’est pas un Ticket Sales tant que Sales ne l’a pas acceptée.

Offlinesync conserve les preuves techniques et les soumissions. Sales contient uniquement les tickets acceptés et alimente stats, caisse, ledger et payout.

Règles non négociables

  • Pas de payout offline.
  • Pas de stats Sales sur une soumission rejetée ou en review.
  • Pas d’auto-accept après résultat connu.
  • Codes offline pré-réservés, signés et bornés.
  • Session fermée = ajustement audité, jamais modification silencieuse.

1. Vue d’ensemble du modèle offline

Le mode offline permet au vendeur d’imprimer une preuve locale, mais la vente devient officielle seulement après synchronisation, validation technique et validation métier.

✅ Ce qui est officiel

Un ticket existe dans sales.ticket uniquement après acceptation par Sales.

  • Compte dans les stats Sales.
  • Impacte la session cash.
  • Peut devenir payable après résultat.
  • Déclenche TicketPlacedEvent.

⛔ Ce qui reste provisoire

Une soumission offline reste dans offlinesync tant qu’elle n’est pas acceptée.

  • Ne compte pas dans les stats Sales.
  • Ne crée pas de payout exposure.
  • N’impacte pas le ledger.
  • Peut être rejetée ou mise en review.
Bleu = client Flutter/POS Cyan = sécurité technique Vert = métier accepté Rouge = risque fraude

6. Implications pour les autres domaines

Le mode offline touche plusieurs domaines, mais chacun garde sa responsabilité propre.

Domaine Responsabilité Impact offline Règle de sécurité
offlinesync Journal technique, grants, codes, submissions. Conserve aussi les rejetés/reviews, payloads bruts et signatures. Ne crée jamais de Ticket Sales.
sales Véritables tickets, argent, lifecycle ticket. Crée les tickets offline acceptés uniquement. Publie TicketPlacedEvent seulement après acceptation.
session Caisse vendeur/session. Normal si session ouverte ; adjustment si session fermée. Jamais de modification silencieuse d’une session fermée.
limitpolicy Limites, block, approval. Deux types de rejet : block limite et rejet sync offline. Block limite peut être REVIEW/REJECT selon policy.
draw / drawresult Cutoff, scheduledAt, resultedAt. Sales vérifie cutoff et refus auto après résultat connu. createdAtDevice n’est jamais preuve suffisante.
catalog.pricing Odds/pricing snapshots. Grant contient snapshot, Sales revalide. Pricing mismatch = no Ticket Sales.
payout Payer les tickets gagnants. Tickets offline pending/review/rejected non payables. Payout offline interdit.
stats/report Rapports métier et risques. Stats Sales séparées des stats Offline Risk. Stats Sales ne lisent jamais offlinesync.
terminal État device, lock, syncState, offline capability. Détermine si POS/mobile peut recevoir un grant. Terminal locked = pas de nouveau grant.
accesscontrol User/permissions/revocation. Valide le vendeur au moment d’émission du grant. Révocation = pas de nouveau grant ; anciennes sync selon policy.

7. Gates, risques et décisions

Il y a deux familles de rejet : le rejet de politique de vente/limite, et le rejet de synchronisation offline.

Rejet LimitPolicy Block

Rejet métier pendant l’évaluation Sales

Exemple : montant trop élevé, vendeur restreint, outlet policy, limite par draw ou par session. 

decision = SALES_REJECTED
reason = LIMIT_POLICY_BLOCKED
sales_ticket_id = null
stats_sales = no
commission = no
Rejet après Sync Offline

Rejet lié au contexte offline

Exemple : sync après résultat, cutoff dépassé, heure device douteuse, code inconnu, séquence replay. 

decision = SALES_REJECTED or TECHNICALLY_REJECTED
reason = SYNC_AFTER_RESULT / INVALID_SIGNATURE / DEVICE_TIME_UNTRUSTED
sales_ticket_id = null
stats_sales = no
Challenge Décision sécuritaire Faille Mobile/POS Correction
Heure device trafiquée createdAtDevice n’est pas une preuve. Mobile peut changer l’heure. Comparer receivedAtServer, cutoff, scheduledAt, resultedAt ; rejet/review après tirage.
Offline plusieurs jours Grant limité, pas d’auto-accept après résultat. Vendeur attend les résultats. AllowedDraws bornés, limites basses, sync tardive = reject/review.
Codes inventés Codes offline pré-réservés. App génère faux codes plausibles. OfflineCodeBatch signé, séquence serveur, statut par code.
Reset app Limites côté serveur, pas local-only. Données locales effacées. last sequence server-side, code batch non réutilisable.
Sync partielle Séquences sans trou pour auto-accept. Vendeur cache les gagnants. Gaps = review/reject, surtout après résultat.
Session fermée Adjustment séparé. Cash de fermeture modifié après coup. SalesSessionOfflineAdjustment audité.
Payout offline Interdit. Ticket pending utilisé pour payout. Payout exige Ticket Sales accepté + winning + not paid.

8. Évolution de Tchalanet et ordre de développement

Le développement doit stabiliser le modèle Sales avant d’ouvrir largement l’offline. La sécurité des données et des stats est prioritaire.

P0

Stabiliser Sales money model

Ajouter stakeAmount, feeAmount, totalAmount, mapper, entity, audit, vues read model, receipt.

Livrable : tickets online propres et events contenant stake/fee/total.

P1

Créer core.offlinesync minimal

OfflineSalesGrant, OfflineCodeBatch, OfflineBatch, OfflineSaleSubmission, gates techniques, endpoints sync.

Livrable : stockage audit sans créer Ticket Sales.

P2

SyncOfflineSalesCommand dans Sales

Gates métier : cutoff, pricing, limits, session, duplicate, money breakdown. Créer Ticket seulement si accepté.

Livrable : flow auto-accept / reject / review.

P3

Session post-close adjustment

Créer le mécanisme d’ajustement de session pour tickets offline acceptés après fermeture.

Livrable : cash reports audités, sans modification silencieuse.

P4

Features Cashier Flutter/POS

Local DB chiffrée, queue offline, receipt offline, sync automatique, statut par ticket, reprint.

Livrable : vendeur peut vendre offline sans polluer Sales.

P5

Tenant Admin + review queue

Dashboard risk, approve/reject submissions, penalties, limites offline par vendeur/outlet/terminal.

Livrable : gouvernance tenant du risque offline.

P6

Super Admin / Ops

Politiques globales, audit investigation, révocation terminal, monitoring multi-tenant, exports.

Livrable : contrôle plateforme et conformité.

Roadmap recommandée : ne pas commencer par l’UI offline. D’abord sécuriser Sales, puis Offlinesync, puis seulement Flutter/POS. Sinon les stats, la session et le payout seront contaminés par des objets provisoires.
