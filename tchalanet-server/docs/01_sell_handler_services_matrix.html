SellTicketCommandHandler & Services Matrix

Ce document explique ce qui change dans le Sell handler et liste les services/endpoints/sécurité autour de ledger, payout, autonomy, limitpolicy, outlet, terminal, session, sales et offlinesync.

1. Ce qui change dans SellTicketCommandHandler

Le handler ne doit plus être un simple create ticket. Il devient le point d’entrée sécurisé pour les ventes online officielles. Les ventes offline passent par SyncOfflineSalesCommandHandler, mais doivent réutiliser les mêmes services de préparation/validation quand possible.

Avant

Handler trop direct

  • Risque de tenant/user depuis request body.
  • Pas de séparation claire stake / fee / total.
  • Events pouvant partir pour tickets non définitivement acceptés.
  • Session/cash pas assez explicite.
  • Offline non modélisé.
Après

Handler orchestrateur sécurisé

  • Contexte via @CurrentContext, jamais depuis le client.
  • Résout terminal, outlet, seller, session.
  • Prépare lignes + pricing snapshot.
  • Évalue limitpolicy/autonomy.
  • Crée Ticket uniquement si vente acceptée ou pending approval métier autorisé.
  • Publie events after-commit seulement quand approprié.

2. Nouveau découpage conseillé dans Sales

Pour ne pas dupliquer online/offline, le handler online et le handler offline doivent partager certains services applicatifs.

SellTicketCommandHandler

Online officiel

Crée un ticket online accepté ou une approval métier online. Ne reçoit jamais de payload offline brut.

SyncOfflineSalesCommandHandler

Offline accepté/rejeté

Transforme une submission techniquement valide en Ticket seulement si tous les gates Sales passent.

TicketCreationService

Factory partagée

Centralise création Ticket, money, publicCode, TicketPlacedEvent payload, invariants. 

SellTicketCommandHandler
  -> TicketSaleContextResolver
  -> TicketLinePreparationService
  -> DrawCutoffRule
  -> LimitPolicyGateway
  -> AutonomyApprovalEvaluator
  -> TicketCreationService
  -> TicketWriterPort.save()
  -> AfterCommit TicketPlacedEvent

SyncOfflineSalesCommandHandler
  -> OfflineSalesGateEvaluator
  -> same TicketLinePreparation / TicketCreationService where possible
  -> TicketWriterPort.save() only if accepted
  -> AfterCommit TicketPlacedEvent

4. Checklist de refactor immédiat du Sell

À faire avant d’ouvrir largement le mode offline dans Flutter/POS.

1

Contexte serveur

Supprimer tenant/user/session depuis request body. Résoudre via @CurrentContext, terminal, session ouverte/grant.

2

Money breakdown

Ajouter stakeAmount, feeAmount, totalAmount. Les payouts utilisent stake, la session utilise total.

3

LimitPolicy block

Si block strict : pas de ticket officiel, pas d’event, pas de stats. Si review : pending workflow clair.

4

Events after-commit

TicketPlacedEvent uniquement pour ventes acceptées. Inclure stake/fee/total, origin, syncStatus, postingMode.

5

Offline séparé

Le handler online ne reçoit pas les submissions offline. Offline passe par offlinesync puis SyncOfflineSalesCommand.
