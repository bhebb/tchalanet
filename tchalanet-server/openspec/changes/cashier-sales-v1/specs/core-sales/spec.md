# Specification Delta — core.sales

## ADDED Requirements

### Requirement: Core sales owns official ticket receipt content

`core.sales` SHALL own the official ticket receipt content and formatters for ticket proof.

`platform.document` SHALL render generic document requests only and SHALL NOT contain ticket-specific formatting rules.

`platform.communication` SHALL deliver text messages only and SHALL NOT contain ticket-specific formatting rules.

#### Scenario: Cashier prints ticket through sales receipt query

- **GIVEN** a cashier requests ticket print
- **WHEN** the feature needs printable content
- **THEN** it SHALL ask `core.sales.api.query.receipt.FormatTicketReceiptPrintQuery`
- **AND** SHALL NOT import `core.sales.internal.*`

### Requirement: Sales exposes receipt API models and queries

`core.sales.api` SHALL expose receipt queries and immutable receipt read models.

#### Scenario: Feature obtains message content

- **GIVEN** a cashier requests send
- **WHEN** the feature needs a message body
- **THEN** it SHALL ask `FormatTicketReceiptMessageQuery`
- **AND** receive `TicketReceiptMessageContent`

### Requirement: Ticket receipt message and backup shareable text use same formatter

`TicketBackupInfo.shareableText` SHALL be generated by the same formatter used by `FormatTicketReceiptMessageQuery`.

#### Scenario: Backup equals system-send body

- **GIVEN** an accepted sell response
- **AND** the same ticket is formatted for system send
- **THEN** `backup.shareableText` SHALL be byte-identical to the system-send message body

### Requirement: Sales evaluates preview and sell through one evaluator

`PreviewTicketSaleQueryHandler` and `SellTicketCommandHandler` SHALL both use `SaleAcceptanceEvaluator`.

#### Scenario: Preview and sell share validation pipeline

- **GIVEN** a basket with invalid selection format
- **WHEN** preview and sell are evaluated
- **THEN** both SHALL report the same issue code and seller instruction

### Requirement: Preview is read-only and final sell rechecks transactionally

Preview SHALL perform no state mutation. Final sell SHALL recheck exposure under lock or atomic upsert before creating a ticket.

#### Scenario: Exposure changes after preview

- **GIVEN** preview returns acceptable
- **AND** another seller consumes remaining exposure before final sell
- **WHEN** final sell rechecks exposure
- **THEN** sell SHALL return `REJECTED` with issue `EXPOSURE_CHANGED`
- **AND** no ticket SHALL be created

### Requirement: Exposure limits are cumulative

Sales SHALL apply exposure limits using existing exposure plus current basket exposure.

#### Scenario: Seller cannot bypass limit by splitting tickets

- **GIVEN** a selection limit of 250 HTG
- **AND** existing exposure for the seller/session/draw/selection is 249 HTG
- **WHEN** the seller attempts to sell another 10 HTG on the same selection
- **THEN** the sale SHALL be rejected or require changes
- **AND** the issue SHALL expose `allowedRemaining = "1.00 HTG"`

### Requirement: Basket lines are grouped before exposure evaluation

Sales SHALL normalize and group basket lines by exposure key before applying limits.

#### Scenario: Multiple lines on same selection exceed limit

- **GIVEN** a selection limit of 250 HTG
- **WHEN** the basket contains two BOLET 45 lines of 150 HTG and 120 HTG
- **THEN** the basket exposure SHALL be treated as 270 HTG
- **AND** the evaluation SHALL require changes

### Requirement: Pending approval is retained in core but not exposed to POS

`SellTicketOutcome.PENDING_APPROVAL` SHALL remain available in core sales for non-POS channels.

Cashier POS SHALL map approval-required decisions to `REQUIRES_CHANGES` / `APPROVAL_REQUIRED`.

#### Scenario: Web channel can still receive pending approval

- **GIVEN** a sale channel that supports approval workflow
- **WHEN** evaluation requires approval
- **THEN** core sales MAY return `PENDING_APPROVAL`

#### Scenario: POS channel maps approval to changes

- **GIVEN** cashier POS sale requires approval
- **WHEN** the POS response is produced
- **THEN** the response SHALL NOT expose `PENDING_APPROVAL`
- **AND** SHALL contain issue `APPROVAL_REQUIRED`

### Requirement: Accepted ticket backup is built before response return

Backup content SHALL be built synchronously after ticket persistence and before returning the sell response.

Domain events SHALL be published after commit.

#### Scenario: Sell accepted returns backup immediately

- **GIVEN** a ticket is accepted and persisted
- **WHEN** the sell response is returned
- **THEN** backup content SHALL be present in the response
- **AND** `TicketPlacedEvent` SHALL be scheduled after commit

### Requirement: Ticket receipt groups lines by game

The official printed ticket receipt SHALL group lines by game.

#### Scenario: Receipt groups BOLET and MARYAJ lines

- **GIVEN** a ticket with BOLET and MARYAJ lines
- **WHEN** the receipt is formatted
- **THEN** BOLET lines SHALL appear under a BOLET section
- **AND** MARYAJ lines SHALL appear under a MARYAJ section

### Requirement: Public code normalization

Sales verification SHALL accept dashed and non-dashed public codes case-insensitively.

#### Scenario: Dashed code verifies same ticket

- **GIVEN** a ticket public code `PSGV4AXJ`
- **WHEN** customer verifies `PSGV-4AXJ`
- **THEN** sales SHALL normalize the code to `PSGV4AXJ`
- **AND** return the same ticket verification view

### Requirement: Money strings are stable

Sales cashier/receipt API models SHALL serialize money as `X.XX CURRENCY` strings.

#### Scenario: Stake serializes with scale 2

- **GIVEN** a stake of 5 HTG
- **WHEN** it is returned in cashier/receipt response
- **THEN** it SHALL be serialized as `5.00 HTG`

### Requirement: Cancel releases exposure

Cashier cancel within the allowed window SHALL reverse or release exposure consumed by the ticket.

#### Scenario: Cancelled ticket frees exposure

- **GIVEN** a sold ticket consumed exposure on BOLET 45
- **WHEN** the ticket is cancelled within the allowed window
- **THEN** exposure SHALL be released or reversed
- **AND** later evaluations SHALL not count the cancelled ticket as active exposure

## MODIFIED Requirements

### Requirement: Sales receipt formatter is not an outbound port

`TicketReceiptFormatter` SHALL NOT live in `core.sales.internal.application.port.out` because it is not an external dependency port.

#### Scenario: Official receipt formatter lives in application receipt package

- **GIVEN** receipt formatting code exists
- **WHEN** it is placed in the codebase
- **THEN** it SHALL live under `core.sales.internal.application.receipt` or `core.sales.internal.application.receipt.formatter`
