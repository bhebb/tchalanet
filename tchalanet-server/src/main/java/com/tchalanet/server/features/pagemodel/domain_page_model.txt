## 2) DOMAIN_PAGE_MODEL.md – Domaine des pages & templates

```markdown
# Domaine pagemodel

> Domaine responsable de la **description, la personnalisation et la résolution** des pages (publiques, privées, admin) à partir de modèles JSON stockés en base.

---

## 1. Rôle du domaine

**Responsabilité principale**

Centraliser la **configuration des pages** Tchalanet (public/privé/admin) sous forme de modèles JSON (PageModel) stockés en base, et fournir au BFF une vue normalisée de ces modèles, enrichie avec la **donnée dynamique** adaptée au tenant, au rôle et à la langue.

**Ce que le domaine fait**

- Décrit la structure des pages (meta, thème, shell, layout, widgets) via un `PageModel`.
- Gère des **templates de page** globaux (`PageTemplate`) qui servent de base pour les tenants.
- Gère des **instances de page par tenant** (`PageModel`), incluant :
  - statuts (DRAFT, PUBLISHED, ARCHIVED)
  - mapping tenant + rôle + identifiant logique de page.
- Résout, pour une requête donnée :
  - quel modèle utiliser (template vs page tenant)
  - dans quelle langue (`currentLang`)
  - quelles sources dynamiques doivent être appelées.
- Expose au BFF une `PageModelResponse` qui contient :
  - `currentLang`, `langs`
  - le `pageModel` normalisé
  - un bloc `dynamic` (données métiers)
  - des informations i18n (`namespaces`, `overrides`)
  - des `issues` (diagnostics de configuration).

Le domaine **pagemodel** ne calcule pas de KPIs métier lui-même : il orchestre des providers dynamiques situés dans d’autres features (`pagemodel.shared.dynamic`, `private_dashboard.dynamic`, etc.).

---

## 2. Cas d’utilisation principaux

### Public

- Afficher la **page publique par défaut** (ex : `public.home`) sans tenant (page Tchalanet globale).
- Afficher la **page publique propre à un tenant** si une page spécifique a été configurée.
- Renvoyer, pour la home publique :
  - les tirages et résultats récents,
  - les plans B2B,
  - les actualités,
  - des sections statiques (hero, features, check ticket…).

### Privé

- Afficher le **dashboard privé** adapté au rôle :
  - `SUPER_ADMIN` → vue plateforme
  - `TENANT_ADMIN` → vue tenant
  - `OPERATOR` → plus tard
  - `CASHIER` → vue vendeur POS
- Résoudre, à partir du `PageModel`, quelles sections doivent être alimentées depuis :
  - `reporting/*`
  - `draw`, `session`, `autonomy`, `audit`
  - etc.
- Permettre l’évolution de la structure des dashboards via la configuration des widgets et du layout (sans changer le code).

### Admin / Template

- Permettre au **super admin** de gérer les templates de pages (`PageTemplate`), par scope :
  - templates publics
  - templates privés par rôle (`TENANT_ADMIN`, `CASHIER`, etc.)
  - templates admin (super admin).
- Permettre au **tenant admin** de :
  - cloner un template de page en une instance `PageModel` pour son tenant,
  - modifier la version DRAFT,
  - publier une version (qui devient la version active),
  - dupliquer / archiver ses modèles.
- Gérer le **bootstrap** d’un nouveau tenant :
  - à la création du tenant, cloner les templates par défaut (public/privé) en pages initiales pour ce tenant.

---

## 3. Frontières & dépendances

**Le domaine pagemodel dépend de :**

- `accesscontrol`
  - Résolution de `TchRole` (SUPER_ADMIN, TENANT_ADMIN, OPERATOR, CASHIER).
- `tenantconfig`
  - Informations par tenant (locale par défaut, paramètres d’affichage).
- `identity` / `user_preference`
  - Langue préférée de l’utilisateur, densité, thème.
- `reporting/*`
  - `salesreport`, `outletperformance`, `tenantkpis`, `tenant_dashboard` pour les dashboards privés.
- `draw`
  - Tirages, résultats, channels, prochains tirages pour les sections draws/results.
- `news`
  - Pour les widgets d’actualités publiques.
- `plans/billing`
  - Pour les widgets de plans B2B publics.
- `autonomy` / `validation`
  - Pour les blocs de validations en attente dans les dashboards privés.
- `audit`
  - Pour les blocs d’activité récente.

**Le domaine pagemodel est utilisé par :**

- Le BFF / adaptateurs REST qui exposent :
  - `GET /api/public/pages/{slug}`
  - `GET /api/private/dashboard` (par exemple)
  - `GET /api/admin/pages/templates`

Le domaine pagemodel **ne** doit pas contenir de logique métier lourde (calcul financier, settlement, etc.). Il orchestre et compose des données provenant d’autres domaines.

---

## 4. Modèle conceptuel

### 4.1. PageTemplate

Un `PageTemplate` représente un modèle de page **global** (non lié à un tenant), utilisé comme base clonable.

Champs conceptuels :

- `id` : UUID
- `logical_page_id` : identifiant logique de la page
  - ex : `PUBLIC_HOME`, `TENANT_ADMIN_DASHBOARD`, `CASHIER_DASHBOARD`
- `scope` : `public` | `private` | `admin`
- `role` : `TchRole` optionnel (pour les pages privées)
- `schema_version` : version du schéma PageModel
- `page_model_json` : JSONB contenant le `PageModel`
- `is_default` : booléen (template par défaut pour ce scope/role)
- métadonnées : `created_at`, `updated_at`, `created_by`, etc.

Seul le **super admin** peut créer/éditer/supprimer des `PageTemplate` (soft-delete).

### 4.2. PageModel (instance tenant)

Un `PageModel` représente une page concrète pour un **tenant** donné.

Champs conceptuels :

- `id` : UUID
- `tenant_id` : UUID (nullable pour les pages globales publiques Tchalanet)
- `logical_page_id` : comme pour les templates
- `scope` : `public` | `private` | `admin`
- `role` : `TchRole` optionnel
- `status` : `DRAFT` | `PUBLISHED` | `ARCHIVED`
- `template_id` : référence vers `PageTemplate` utilisé à l’origine (optionnel)
- `page_model_json` : JSONB
- métadonnées : `version`, `created_at`, `updated_at`, `updated_by`, `deleted_at`.

Règle importante :
> Pour une combinaison `(tenant_id, logical_page_id, role)`, au plus **un seul** `PageModel` peut être en statut `PUBLISHED`.

Ce `PageModel` contient les blocs :

```jsonc
{
  "meta": {
    "id": "public.home",
    "scope": "public",
    "slug": "home",
    "context": "public_home",
    "schema_version": 1,
    "langs": ["fr", "en", "ht"],
    "default_lang": "fr"
  },
  "theme": {
    "preset": "tchalanet_default",
    "mode": "light",
    "density": 0
  },
  "shell": {
    "header": { ... },
    "sidenav": null,
    "footer": { ... }
  },
  "content": {
    "layout": {
      "component": "GridLayout",
      "rows": [
        {
          "id": "hero",
          "label_key": "layout.public_home.hero",
          "columns": [
            { "span": 12, "widgets": ["home.hero"] }
          ]
        }
        // ...
      ]
    },
    "widgets": {
      "home.hero": {
        "type": "HeroWidget",
        "binding": { "mode": "static" },
        "props": { ... }
      },
      "home.draws": {
        "type": "DrawsWidget",
        "binding": { "mode": "dynamic", "source": "results_by_game" },
        "props": { ... }
      }
      // ...
    }
  }
}

4.3. Statuts et versions

DRAFT : version en édition, non visible des utilisateurs finaux.

PUBLISHED : version active (unique par (tenant, logical_page_id, role)).

ARCHIVED : anciennes versions stockées pour historique/rollback.

Workflow typique :

Cloner un PageTemplate → créer un PageModel en DRAFT.

Modifier le JSON (layout, widgets, props).

Publier → status PUBLISHED (et eventuelle ancienne version passe en ARCHIVED).

Optionnel : dupliquer un modèle existant pour expérimenter une nouvelle version.

5.2. Contraintes

Index unique sur (tenant_id, logical_page_id, role, status='PUBLISHED').

RLS pour page_model : visibilité limitée au tenant_id correspondant (sauf super admin).

6. API & interaction BFF
6.1. Réponse type PageModelResponse

Pour chaque page, le BFF renvoie une réponse structurée :

{
  "currentLang": "fr",
  "langs": ["fr", "en", "ht"],

  "pageModel": { /* PageModel normalisé */ },

  "dynamic": {
    // données dynamiques injectées selon les bindings
  },

  "i18n": {
    "namespaces": ["nav", "layout", "cta", "home", "footer", "legal"],
    "overrides": {
      // éventuels overrides par tenant+locale
      "home.hero.title": "Titre spécifique tenant"
    }
  },

  "issues": [
    // diagnostics optionnels (widget sans source dynamic, type inconnu, clé i18n manquante…)
  ]
}

6.2. Résolution du PageModel

Pour une requête :

Public (non authentifié) :

Si URL spécifique à un tenant → chercher un PageModel PUBLISHED pour ce tenant et logical_page_id donné.

Sinon → utiliser le modèle global tenant_id = NULL.

Privé (authentifié) :

Résoudre tenant_id et TchRole effectif.

Chercher PageModel PUBLISHED pour (tenant, logical_page_id, role).

Si absent → fallback sur le PageTemplate par défaut.

7. Résolution de langue & i18n

La langue courante n’est pas stockée dans PageModel.
Elle est calculée à chaque requête selon l’algorithme :

Langue forcée par URL / paramètre (/fr/... ou ?lang=en).

Langue préférée de l’utilisateur (user_preference.locale, si connecté).

Langue par défaut du tenant (tenant.default_locale).

Fallback :

meta.default_lang si présent,

sinon premier élément de meta.langs.

Pseudo-code :

String resolveCurrentLang(Request request, Tenant tenant, UserPreference userPref, PageModel pageModel) {
    var allowed = pageModel.meta().langs();

    var langFromUrl = extractLangFromUrlOrQuery(request);
    if (allowed.contains(langFromUrl)) return langFromUrl;

    if (userPref != null && allowed.contains(userPref.locale())) return userPref.locale();

    if (tenant.defaultLocale() != null && allowed.contains(tenant.defaultLocale()))
        return tenant.defaultLocale();

    if (pageModel.meta().defaultLang() != null && allowed.contains(pageModel.meta().defaultLang()))
        return pageModel.meta().defaultLang();

    return allowed.get(0);
}


Les overrides i18n par tenant sont gérés via une table i18n_override séparée, liée au tenant et à la locale.

8. Sécurité, multi-tenant & templates

Les PageTemplate sont gérés par le super admin uniquement.

Les PageModel peuvent être gérés par :

super admin (tous tenants)

tenant admin (son tenant uniquement)

RLS s’applique sur tenant_id pour page_model.

Lors de la création d’un nouveau tenant :

un service de bootstrap clone les PageTemplate par défaut en PageModel DRAFT ou PUBLISHED pour ce tenant (public + privés par rôle).

L’adapter HTTP doit vérifier :

que l’utilisateur a le droit d’accéder à la page demandée (via accesscontrol),

que le PageModel utilisé correspond bien à son tenant et son rôle.

9. Domaines existants (référence)

À titre indicatif, les domaines actuellement présents dans Tchalanet :

accesscontrol — permissions & rôles par tenant.

audit — audit applicatif & révisions.

draw — tirages & résultats.

ticket — création & paiement de tickets.

session — sessions POS & vendeurs.

tenantconfig — configuration de tenant (limites, odds, etc.).

pagemodel — configuration dynamique des pages publiques/privées (ce domaine).

identity — utilisateurs & profils (hors auth Keycloak).

reporting/* — rapports agrégés par tenant, outlet, jeu, période.

Le domaine pagemodel doit rester un orchestrateur de configuration et de composition, en cohérence avec ces domaines, sans dupliquer de logique métier critique.
