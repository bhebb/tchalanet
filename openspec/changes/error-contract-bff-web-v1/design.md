# Design — error-contract-bff-web-v1

## Server contract

The server already has two response channels:

- blocking errors: `ProblemDetail` generated by `GlobalErrorHandler`;
- successful or partial responses: `ApiResponse<T>` enriched by `ApiResponseBodyAdvice`,
  `ApiResponseContext`, `ApiNotice`, and `ServiceStatus`.

BFF orchestration must choose the channel intentionally for every slice call.

This is a clarification of the existing contract, not a replacement:

```text
2xx JSON -> ApiResponse<T>
4xx/5xx -> ProblemDetail
```

The canonical existing documentation is:

```text
tchalanet-server/docs/conventions/api/api_response.md
tchalanet-server/openspec/context/76-api-response-rules.md
```

This change should refine that documentation instead of introducing a parallel mechanism.

## Blocking slice failure

A blocking slice failure stops the BFF response.

The server returns `ProblemDetail` with:

- HTTP status matching the failure;
- stable `code`;
- `timestamp`, `path`, `method`;
- `requestId` when present;
- `traceId`/`spanId` when tracing is present;
- generated `errorId`.

The web translates by `code`. It uses status/category fallback only when the code is unknown.
Raw backend `title`, `detail`, exception messages, provider messages, SQL messages, and stack traces
are not public/minimal UI copy.

## Non-blocking slice failure

A non-blocking slice failure preserves the primary response body and adds a notice/service status.

The response remains `2xx` and uses:

- `ApiResponse.status = SUCCESS_WITH_WARNINGS`, `PARTIAL`, or `PENDING`;
- `ApiNotice.code` as a stable machine code;
- `ApiNotice.severity` as `INFO`, `WARN`, or `ERROR`;
- `ApiNotice.domain` or source as the owning slice;
- `ApiNotice.meta` for support correlation:
  - `requestId`;
  - `traceId`;
  - `spanId`;
  - `errorId`;
  - optional `slice` or `operation`.

Example code shape:

```text
platform.identity.activation.error
platform.identity.activation.warning
core.sales.sale_approval_required
features.dashboard.stats.partial
```

The existing `ServiceStatus` channel remains for downstream service health, but it is not enough for
user copy by itself. A user-visible partial failure should include an `ApiNotice` with a stable code.
These notices are non-blocking feedback. They must not be converted into HTTP failures by clients.

## Generic notice shape

`ApiNotice` should stay the single user-feedback primitive for 2xx responses:

```json
{
  "code": "platform.identity.activation.error",
  "message": "Identity activation could not be completed.",
  "domain": "platform.identity",
  "severity": "WARN",
  "meta": {
    "source": "identityActivation",
    "service": "keycloak",
    "operation": "completeFirstLogin",
    "requestId": "tch_req_...",
    "traceId": "4bf92f3577b34da6a3ce929d0e0e4736",
    "spanId": "00f067aa0ba902b7",
    "errorId": "6bfa..."
  }
}
```

Reserved `meta` keys:

| Key | Meaning |
|---|---|
| `source` | Fine-grained emitter or slice inside the domain |
| `service` | Downstream service/provider when relevant |
| `operation` | Stable operation name useful for support |
| `requestId` | Request correlation id |
| `traceId` | Distributed trace id |
| `spanId` | Distributed span id |
| `errorId` | Server-generated error id for the caught optional failure |

Domain-specific metadata can be added, but frontend behavior must not depend on many bespoke keys.

## Generic server helper

To keep the approach intuitive, application code should not build the full `meta` map manually every
time. The implementation should provide one small helper/factory around `ApiResponseContext`, for
example:

```java
ApiResponseNotices.warn(
    "platform.identity.activation.error",
    "Identity activation could not be completed.",
    "platform.identity",
    NoticeSource.of("identityActivation")
        .service("keycloak")
        .operation("completeFirstLogin")
        .error(ex)
);
```

The helper should:

- set `code`, `message`, `domain`, and `severity`;
- attach `source`, `service`, and `operation` when provided;
- attach `requestId`, `traceId`, `spanId`, and generated `errorId`;
- add the notice to `ApiResponseContext`;
- optionally add `ServiceStatus` when the failure is a downstream service degradation.

Controllers and BFF services then only choose:

```text
blocking? throw stable exception / ProblemDetail
non-blocking? ApiResponseNotices.warn/error/info(...)
```

That keeps the `ApiResponseBodyAdvice` design as the central contract and avoids per-feature
boilerplate.

## Concrete BFF examples

### Optional identity activation slice fails

```java
try {
    activationView = identityActivation.activateIfNeeded(ctx);
} catch (Exception ex) {
    ApiResponseNotices.warn(
        PlatformIdentityNoticeCodes.ACTIVATION_ERROR,
        "Identity activation could not be completed.",
        "platform.identity",
        NoticeSource.of("identityActivation").service("keycloak").operation("activateIfNeeded").error(ex)
    );
    activationView = IdentityActivationView.unavailable();
}

return dashboardView.withActivation(activationView);
```

HTTP response stays 2xx and the web shows a warning translated from
`platform.identity.activation.error`.

### Required sales slice fails

```java
var sale = commandBus.execute(command);
return ApiResponse.success(mapper.toSellResponse(sale));
```

If the command fails with a stable `TchException`, `GlobalErrorHandler` returns `ProblemDetail`.
The web translates the exact error code and shows the support reference from trace fields.

### Optional provider health degrades

```java
ApiResponseNotices.warn(
    "features.dashboard.provider_results.degraded",
    "Some provider results are temporarily unavailable.",
    "features.dashboard",
    NoticeSource.of("providerResults").service("uslottery").operation("latestResults").error(ex)
);
ApiResponseContext.get().addServiceStatus("uslottery", ServiceHealth.DEGRADED, "Latest results unavailable");
```

The response becomes `PARTIAL` because a service is degraded and includes a notice the web can
translate and display.

## Central code catalog

Backend code strings should be centralized close to the owner:

- `common` can own transversal codes such as `request.not_readable`, `validation.failed`,
  `access.denied`, `internal.unexpected`;
- each domain/platform/feature slice owns its namespaced codes in an API-facing constants/catalog
  type;
- controllers and BFF services use those constants instead of ad hoc string literals.

Code naming:

```text
<layer-or-domain>.<capability>.<operation>.<condition>
```

Codes are stable API contract. Renaming a code is a breaking change for web i18n and support docs.

Prefer lowercase dotted codes for new API-facing error/notice codes. Legacy uppercase codes may be
mapped by the web normalizer during migration, but new BFF work should use the dotted convention.

This is centralized by ownership, not as one huge global enum. The API-facing constants should make
the common case obvious:

```java
throw new TchBusinessRuleException(
    PlatformIdentityErrorCodes.ACTIVATION_BLOCKED,
    "Identity activation is blocked"
);
```

Blocking flow:

```text
TchException(code)
  -> GlobalErrorHandler
  -> ProblemDetail.code + requestId/traceId/spanId/errorId
  -> Web exact-code translation
```

Non-blocking flow:

```text
ApiResponseNotices.warn/error/info(code, message, domain, source)
  -> ApiResponseContext
  -> ApiResponseBodyAdvice
  -> ApiResponse.notices + services
  -> Web exact-code translation
```

The caller should normally pass only the stable code, safe fallback message, domain, source/service,
severity, and optional exception. The infrastructure attaches trace/support metadata.

## Web translation policy

Frontend resolution order:

1. known exact code translation;
2. category translation inferred from code/status/severity;
3. generic fallback translation.

Fallback exists for resilience, not as the normal state for known product errors.

Translation key examples:

```text
common.errors.codes.platform.identity.activation.error.title
common.errors.codes.platform.identity.activation.error.message
common.errors.categories.service_unavailable.title
common.errors.categories.service_unavailable.message
```

## Trace and support reference

Both blocking errors and non-blocking notices must produce a support reference when correlation
fields are available.

The web should display the support reference according to shell verbosity and copy only safe fields:

- code;
- category;
- severity;
- route;
- source/domain;
- requestId;
- traceId;
- spanId;
- errorId;
- timestamp.

No stack trace, token, request body, response body, or user-entered personal data is copied.

## BFF aggregation policy

Each BFF endpoint should document its slice dependency policy:

```text
required slices -> fail request with ProblemDetail
optional slices -> return data with ApiNotice/ServiceStatus
background slices -> return data and expose async status/pending notice
```

The policy belongs to the BFF/application service, not the controller.
